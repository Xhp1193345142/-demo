# 开发规范

## 目标

这份文档用于约束本项目的开发方式，保证随着功能变复杂，代码仍然容易理解、容易维护、容易继续迭代。

## 通用原则

- 优先做小步、连续的迭代，不做大而全的重写。
- 产品规则要在代码和文档里写清楚，避免只靠口头约定。
- 不要把“临时能跑”的逻辑混进公共基础层。
- 代码要以“几周后还能快速看懂”为目标来写。

## 分支与开发流程

- 所有开发都从项目根目录 `/Users/haopu_xi/project/Buyu` 开始。
- 尽量保持“一条功能线一个分支”或“一个问题一个修复分支”。
- 大功能开始前，先确认数据模型和用户流程。
- 每次功能完成后至少要做三件事：
  - 执行对应检查
  - 如果行为变化了，同步更新文档
  - 把后续待做事项写清楚

## 后端规范

### App 职责边界

- `accounts`：注册用户与基础资料
- `core_api`：基础 API 和健康检查接口

不要把不相关的业务逻辑随意塞进 `core_api`。

### 模型规范

- 核心业务规则尽量放在模型附近，或者明确的 service 模块中。
- 任何表结构调整都必须配套 migration。

### 视图与 API 规范

- 视图尽量保持轻薄。
- 注册、登录和权限校验必须放在后端，不只放前端。
- API 返回结构要稳定、可预期。
- 当接口数据结构开始变复杂时，优先使用 serializer，不要手写很长的字典拼装逻辑。

### Admin 规范

- Django admin 是项目运营流程的一部分，不是可有可无的附属功能。
- 关键模型都应该注册到 admin。
- `list_filter`、`search_fields` 等配置要尽量实用，方便后续运营与排查。

## 前端规范

### 路由规范

- 使用 App Router 约定。
- 优先使用“数据驱动页面模板”，避免一页一页重复硬写。
- 公共展示页和登录页要保持边界清晰。

### 组件规范

- 优先写小而可复用的组件。
- 可复用的静态结构优先放到 `src/lib`。
- 不要把复杂业务判断和样式结构混在一大段 JSX 里。

### 数据获取规范

- 公开内容优先采用服务端渲染或路由级获取。
- 没有交互必要时，不要过早引入客户端请求。
- 内容权限判断不能只写在前端。

### 样式规范

- 除非明确要改视觉方向，否则应保持当前视觉系统一致。
- 优先使用共享 CSS 变量维护颜色和字体体系。
- 避免随手加大量一次性样式，导致界面逐渐失控。

## 命名规范

- 代码里的变量、函数、类、字段命名统一使用英文。
- 用户可见文案可以使用中文。
- 优先使用有业务语义的命名，不要用模糊通用词。

推荐示例：

- `display_name`
- `phone`
- `last_login`

不推荐示例：

- `level2`
- `dataInfo`
- `tempStatus`

## 文档更新规范

当以下内容变化时，必须同步更新文档：

- 数据模型
- 鉴权流程
- 环境配置
- 开发命令

当前至少要保持最新的文档有：

- `README.md`
- 本文件
- `docs/testing-standards.md`
- `agent.md`

## 完成定义

一个功能在以下条件满足前，不算真正完成：

- 核心行为可正常运行
- 本地检查通过
- 明显边界情况已经处理
- 相关文档已更新
- 如果还有延期项，已经明确写出后续处理方向
