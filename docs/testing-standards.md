# 测试规范

## 目标

本项目的测试目标不是追求数量，而是优先保护最关键的业务规则，让项目在持续迭代时仍然可控。

## 测试优先级

优先测试高风险逻辑：

1. 注册与登录流程
2. 用户资料读写行为
3. Django 后台登录与权限行为
4. 鉴权与会话行为

## 当前阶段说明

项目目前还处于脚手架和基础用户阶段，不需要一开始就铺大量测试，但每新增一条关键业务规则，最好至少留下一个测试。

## 后端测试要求

### 必测内容

- 用户注册校验逻辑
- API 权限控制
- 用户资料更新行为
- Django 会话登录行为

### 推荐结构

- 模型测试可先写在各 app 的 `tests.py`
- API 测试使用 Django 或 DRF 的测试客户端
- 支付相关逻辑独立出来后，再补 service 层测试

### 后端测试示例

- 相同邮箱不能重复注册
- 未登录用户不能访问需要鉴权的接口
- 已登录用户可以读取并更新自己的资料
- Django 后台用户可以正常登录 admin

## 前端测试要求

### 当界面开始变动态后，优先测试这些内容

- 登录表单行为
- 注册表单行为
- 用户资料页展示与编辑

### 后续建议工具

- 单元/组件测试：React Testing Library + Vitest 或 Jest
- 端到端测试：Playwright

在前端还没有复杂交互前，不需要急着把这些工具全部装上。

## 手动验收清单

在合并有实质影响的功能前，至少手动确认：

- 首页可以正常打开
- 登录页可以正常打开
- API 健康检查接口可用
- Django 后台仍然能进入
- migration 能正常执行
- 注册用户数据可以正常写入数据库

## 最低命令检查清单

后端改动较多时，至少执行：

```bash
make check
make migrate
make api-health
```

前端改动较多时，至少执行：

```bash
make lint
make build
```

补充说明：

- 当前 `make build` 使用的是 `webpack`，因为当前环境下 Turbopack 会受到沙箱限制

## 分阶段测试策略

### 阶段 1：脚手架与 MVP

- 以冒烟检查和关键业务规则测试为主
- 不要过早投入大量 UI 快照测试

### 阶段 2：课程与内容

- 增加课程可见性和内容接口测试
- 增加课程列表页和详情页的组件测试

### 阶段 3：支付与订阅

- 再补支付、订阅和权限升级相关测试

## 判断原则

只要一个问题会影响钱、权限、或内容可见性，就值得补自动化测试。
