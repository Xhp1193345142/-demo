# Buyu 协作说明

## 目标

本文件用于约束在 `Buyu` 项目中协作开发时的基础规则，避免后续实现风格混乱、注释语言不一致、目录职责漂移。

## 项目定位

- 这是一个 `Next.js + Django` 的个人网站项目。
- 当前阶段先聚焦用户注册、登录和个人主页展示。
- 课程、支付、会员等能力会在后续阶段逐步补充。

## 代码协作规则

### 1. 注释语言

- 项目中的代码注释统一使用中文。
- 只有在以下情况可以保留英文：
  - 第三方库规定的固定字段
  - 官方错误信息原文
  - 必须与外部接口保持一致的协议字段

推荐做法：

- 复杂业务判断前写简短中文注释
- 容易误解的数据流转处写中文注释
- 支付、权限、会员等级这类高风险逻辑优先补注释

不推荐：

- 写大量无意义注释
- 用注释重复代码表面含义
- 中英文混杂导致阅读割裂

### 2. 命名规则

- 代码变量、函数、类、文件名使用英文。
- 用户可见文案可以使用中文。
- 数据库字段和 API 字段保持英文命名。

示例：

- 好：`display_name`、`phone`、`last_login`
- 不好：`vipLevel2`、`dataInfo`、`tempValue`

### 3. 目录职责

- `frontend/` 负责前端页面、组件、展示层逻辑
- `backend/accounts/` 负责注册用户与基础资料
- `backend/core_api/` 负责轻量 API 和基础接口

不要把不相关的业务逻辑随意塞进公共目录。

### 4. 开发方式

- 优先做小步迭代，不做一次性大重构。
- 每次改动尽量围绕一个明确目标。
- 改完后至少做对应的本地检查。

常用命令：

```bash
make dev
make backend-dev
make frontend-dev
make migrate
make lint
make check
```

### 5. 测试要求

- 涉及注册、登录、会话、用户资料的逻辑，优先补测试。
- 没有测试时，至少补清楚手动验证步骤。

### 6. 修改文档的时机

当以下内容发生变化时，应同步更新文档：

- 鉴权方式
- 数据模型
- 开发命令

## 当前最重要的工程约束

- 代码注释统一使用中文
- 变量与接口字段统一使用英文
- 登录与会话逻辑放后端，不只放前端
- 先做 MVP，再扩复杂功能
